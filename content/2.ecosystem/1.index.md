# Ecosystem

## Agent Gatekeeping

Three tools form the core of how agents interact with the world through OpenApe. Together, they ensure that agents can only do what humans have explicitly approved.

```
┌─────────────────────────────────────────────────────┐
│                Agent Gatekeeping                     │
│                                                      │
│   grapes            shapes             escapes       │
│   ──────────        ──────────         ──────────    │
│   Grant System      Execution Layer    Privilege     │
│                                        Escalation    │
│                                                      │
│   Request, approve  Execute any CLI,   For actions   │
│   and manage        constrained by     the agent     │
│   grants            grants + registry  should never  │
│                                        normally do   │
│                                                      │
│   "May I?"          "Do it             "Exceptionally,│
│                      (within bounds)"   with approval"│
└─────────────────────────────────────────────────────┘
```

### [grapes](/ecosystem/grapes) — The Grant System

The foundation. Without a grant, nothing happens. Grapes handles the complete grant lifecycle: requesting, approving, denying, revoking, and delegating. It's the CLI that both agents and humans use to interact with the OpenApe permission system.

```bash
grapes request "systemctl restart nginx" --audience escapes --wait
grapes approve <grant-id>
grapes run escapes "apt-get upgrade" --approval timed --duration 1h
```

### [shapes](/ecosystem/shapes) — The Execution Layer

The normal path for agent operations. Shapes wraps existing CLI tools (kubectl, aws, gh, az, etc.) and maps their commands to structured permissions. Instead of granting blanket access, each command is parsed into a resource chain and evaluated against grants.

The [Shapes Registry](https://github.com/openape-ai/shapes-registry) is the catalog of available adapters — it defines which tools can be wrapped and how their commands map to permissions.

```bash
shapes explain -- gh repo list myorg        # What permissions does this need?
shapes request -- gh issue create --repo myorg/myrepo  # Request grant + execute
```

### [escapes](/ecosystem/escapes) — Privilege Escalation

The exception path. When an agent needs to do something it should **never** normally have rights for — installing software for another user, modifying system configuration, accessing restricted resources — `escapes` provides controlled, audited privilege elevation. **Es**calated privileges, **with apes**.

```bash
escapes --grant <jwt> -- apt-get install -y nginx
escapes --run-as deploy --grant <jwt> -- systemctl restart app
```

## Developer SDK & Modules

For developers building applications on top of OpenApe.

### Architecture

```
┌──────────────────────────────────────────┐
│          Framework Modules               │
│   nuxt-auth-idp          nuxt-auth-sp    │
├──────────────────────────────────────────┤
│          Protocol Packages               │
│      @openape/auth    @openape/grants    │
├──────────────────────────────────────────┤
│            Foundation                    │
│            @openape/core                 │
├──────────────────────────────────────────┤
│       Agent Infrastructure               │
│     @openape/proxy    @openape/browser   │
└──────────────────────────────────────────┘
```

### Use Cases

| I want to... | Use |
|---|---|
| Add login to my app | [`nuxt-auth-sp`](/ecosystem/nuxt-auth-sp) |
| Run my own IdP | `nuxt-auth-idp` ([config](/reference/idp-configuration)) |
| Build a non-Nuxt integration | `@openape/auth` + `@openape/grants` |
| Control agent HTTP traffic | [`@openape/proxy`](/ecosystem/proxy) |
| Automate browser tasks with grants | [`@openape/browser`](/ecosystem/browser) |

## Design Principles

1. **Separation** — Auth ≠ Grants. Not every app needs both.
2. **Layered** — Core → Protocol → Framework → Agent Tools
3. **Default deny** — No grant = no access. Agents start with zero permissions.
4. **Passkeys-only** — No passwords. NIS2 compliant by design.
5. **Auditable** — Every action traceable: who requested, who approved, what happened.
6. **Minimal tokens** — AuthN says *who*, AuthZ says *what may they do*.
