# @openape/shapes

Adapters that make existing CLI tools grant-aware without modifying their source. Instead of granting agents blanket access to `kubectl` or `aws`, Shapes maps CLI commands to structured permissions and requests grants automatically.

## Installation

```bash
npm install -g @openape/shapes
```

## Quick Start

```bash
# Search for an adapter
shapes adapter search kubectl

# Install an adapter
shapes adapter install kubectl

# Explain what permissions a command needs
shapes explain "kubectl delete pod nginx -n production"
# → Requires: kubernetes:delete:pod (namespace=production, pod=nginx)

# Execute with grant interception
shapes execute "kubectl delete pod nginx -n production"
# → Requests grant, waits for approval, then executes
```

## Concepts

### Adapters

An adapter wraps a specific CLI tool (kubectl, aws, gcloud, terraform, etc.) and understands its command structure. Each adapter:

- Parses the CLI's arguments and flags
- Maps commands to **operations** (structured permission descriptions)
- Builds grant requests with proper resource chains
- Executes the command after grant approval

### Operations

Operations are structured descriptions of what a CLI command does:

```json
{
  "type": "openape_cli",
  "cli_id": "kubectl",
  "operation_id": "delete_pod",
  "resource_chain": [
    { "resource": "namespace", "selector": { "name": "production" } },
    { "resource": "pod", "selector": { "name": "nginx" } }
  ],
  "action": "delete",
  "risk": "high"
}
```

### Resource Chains

Hierarchical resource selectors that describe what's being accessed:

```
cluster → namespace → pod
repository → branch → file
project → environment → service
```

This allows approvers to see exactly what an agent wants to do, not just the raw command string.

### Capabilities

High-level permissions mapped to CLI operations:

```
kubernetes:delete:pod     → kubectl delete pod
kubernetes:get:pod        → kubectl get pod
aws:s3:put-object         → aws s3 cp
github:issues:create      → gh issue create
```

## Commands

### `shapes adapter`

Manage CLI adapters:

```bash
shapes adapter search <query>     # Search the registry
shapes adapter install <name>     # Install an adapter
shapes adapter list               # List installed adapters
shapes adapter info <name>        # Show adapter details
```

### `shapes explain`

Show what permissions a command requires without executing it:

```bash
shapes explain "kubectl delete pod nginx -n production"
# Output:
# CLI: kubectl
# Operation: delete_pod
# Resources: namespace/production → pod/nginx
# Risk: high
# Required approval: once
```

### `shapes execute`

Run a command with grant interception:

```bash
shapes execute "kubectl delete pod nginx -n production"
# 1. Parses command via kubectl adapter
# 2. Builds structured grant request
# 3. Submits to IdP, waits for approval
# 4. Executes command after approval
```

### `shapes request`

Request a capability-based grant:

```bash
shapes request --capability "kubernetes:delete:pod" --approval once
```

## Integration with grapes

`grapes request-capability` uses Shapes adapters under the hood:

```bash
grapes request-capability kubectl \
  --resource pods \
  --selector default \
  --action delete \
  --approval once \
  --wait
```

## Writing Custom Adapters

Adapters are TypeScript modules that implement the `ShapesAdapter` interface:

```typescript
interface ShapesAdapter {
  meta: AdapterMeta
  parse(argv: string[]): ResolvedCommand
  operations: ShapesOperation[]
}
```

An adapter needs to:
1. Define metadata (CLI name, version, description)
2. Parse the CLI's argument structure
3. Map parsed commands to operations with resource chains
4. Define risk levels for each operation

Adapters are published to the Shapes registry and can be installed via `shapes adapter install`.
